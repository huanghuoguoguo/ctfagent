# 第一阶段路线决策：以 Claude Code 对话解题为核心

更新时间：2026-03-28

## 结论

当前仓库应明确采用：

- **Phase 1：`Claude Code` 对话驱动解题**
- **扩展优先：`Skills` 和 `MCP`**
- **Phase 2：再进入 Python 程序驱动**

这不是“先偷懒，后补架构”，而是更适合当前阶段的工程策略。

## 为什么当前阶段应该这样做

现阶段的核心问题不是“如何调度很多 Agent”，而是：

1. `Claude Code` 能不能稳定理解你的 CTF 工作方式
2. 哪些动作真的值得沉淀成 `Skill`
3. 哪些能力真的值得抽成 `MCP`
4. 哪些知识应该进入 `knowledge/`
5. 哪些流程在真实打题时会反复出现

如果这些问题还没被验证，就直接进入程序驱动，会出现两个常见问题：

- 把错误的工作流自动化
- 在 orchestration、状态管理、日志系统上投入很多时间，但解题能力本身并没有真正变强

所以第一阶段最合理的选择，是把 `Claude Code` 当作“交互式 CTF 操作台”，先把真实解题循环跑顺。

## 第一阶段的工作重心

这一阶段应优先建设：

1. `Skills`
2. `MCP`
3. 本地靶场和回归用例
4. 知识沉淀与 exploit pattern
5. 统一的 solver workflow

这一阶段不应优先建设：

1. Python 宿主程序
2. 复杂任务编排
3. 自动批量跑题
4. 自动提交平台
5. 复杂 UI

换句话说，当前阶段优化的是**解题能力的可复用性**，不是**系统自动化程度**。

## 为什么优先扩展 `Skills`

`Skills` 适合承载：

- 题型 SOP
- 决策顺序
- 常见分支
- 低成本验证方法
- 常见误区

它最适合沉淀“CTF 选手的思考方式”，例如：

- 先验证再升级 payload
- 先找源码和元数据，再考虑爆破
- blank page 不能直接当作空站，要考虑 backdoor
- 本地文件读、localhost pivot、短参数 webshell 都应该作为独立模式被识别

这些内容直接影响解题质量，而且修改成本低、迭代快。

## 为什么优先扩展 `MCP`

`MCP` 适合承载：

- 扫描器
- 二进制分析工具
- 平台 API
- 沙箱执行器
- 统一包装的高风险工具

也就是说，`MCP` 更像“CTF 工具接线层”。

Phase 1 里不需要一次把所有工具都接完。只需要把最常用、最确定会反复用的能力先接出来。

例如：

- Web：`nmap`、`ffuf`、`sqlmap`
- Pwn：`checksec`、`gdb`、`pwntools`
- Rev：`strings`、`rizin`、`radare2`
- Forensics：`binwalk`、`exiftool`

## 当前仓库已经在正确方向上

当前仓库已经有一条清晰的第一阶段骨架：

- `ctf-solver-profile`：前置 solver workflow
- `web-ssrf-to-rce-triage`：具体 web exploit skill
- `ctf-knowledge-capture`：本地知识沉淀
- `knowledge/`：writeup 与 pattern
- `targets/`：本地靶场

这说明当前仓库已经从“纯规划”进入“可验证工作流”阶段。

所以接下来最值得做的，不是推翻路线，而是继续沿着这条线补齐：

- 更多题型 skill
- 更实用的 MCP
- 更系统的回归靶场
- 更结构化的知识库

## 第一阶段应该如何定义成功

当下面这些条件成立时，说明第一阶段跑顺了：

1. 面对常见题型，`Claude Code` 能在对话中完成多步分析与验证
2. 你能明确看出哪些流程适合做成 `Skill`
3. 你能明确看出哪些工具适合做成 `MCP`
4. 解题过程可以稳定沉淀到 `knowledge/`
5. 本地靶场可以用于回归验证 skill

只有在这些能力稳定之后，程序驱动才有意义。

## 什么时候进入第二阶段

只有出现以下信号，才应该进入 Python 程序驱动：

1. 你已经有多套稳定的 `Skill`
2. 你已经明确需要批量处理题目或统一管理状态
3. 你已经明确需要平台接入、自动提交或回放日志
4. 你已经知道哪些 challenge 数据需要结构化
5. 你已经能区分“解题逻辑问题”和“编排系统问题”

如果这些信号还不明显，就继续打磨第一阶段。

## 当前建议

当前阶段的默认路线应固定为：

1. 用 `Claude Code` 直接对话解题
2. 通过 `ctf-solver-profile` 约束思考方式
3. 按题型调用具体 `Skill`
4. 缺失能力优先补 `Skill` 或 `MCP`
5. 解完后用 `ctf-knowledge-capture` 沉淀
6. 用 `targets/` 做回归验证

这是当前最利于开发、迭代、验证真实解题能力的路线。
